# coding-rules.md

코딩 프로젝트에서 에이전트가 따르는 행동 규칙이다.
일반 LLM 코딩 실수를 줄이기 위한 지침이며, 과도한 코드 생성보다 신중함을 우선한다.

## 1. 코딩 시작 전

구현 전에 다음을 수행한다.

- 가정을 명시한다. 불확실하면 먼저 묻는다.
- 해석이 여러 가지인 경우 제시하고 선택을 요청한다. 임의로 고르지 않는다.
- 더 단순한 접근이 있으면 먼저 제안한다.
- 모호한 부분이 있으면 멈추고 무엇이 불명확한지 명시한다.

## 2. 단순성 우선

요청된 것만 구현한다.

- 요청하지 않은 기능을 추가하지 않는다.
- 단일 사용 코드에 추상화를 도입하지 않는다.
- 요청하지 않은 유연성·확장성·설정 기능을 넣지 않는다.
- 발생 불가능한 시나리오에 대한 오류 처리를 추가하지 않는다.
- 200줄로 쓸 수 있는 것을 50줄로 줄일 수 있다면 줄인다.

## 3. 최소 변경

요청된 범위만 건드린다.

- 요청과 무관한 인접 코드, 주석, 포맷을 수정하지 않는다.
- 기존 스타일이 다르더라도 그 스타일을 따른다.
- 관계없는 데드 코드를 발견하면 언급하되 삭제하지 않는다.
- 내 변경으로 인해 사용되지 않게 된 import·변수·함수는 제거한다.
- 변경된 모든 줄이 요청에 직접 연결되어야 한다.

## 4. 목표 기반 실행

성공 기준을 먼저 정의한다.

- "검증 추가" → "잘못된 입력에 대한 테스트 작성 후 통과시키기"
- "버그 수정" → "재현하는 테스트 작성 후 통과시키기"
- 여러 단계 작업은 간략한 계획을 먼저 제시한다.

```text
1. [단계] → 검증: [확인 방법]
2. [단계] → 검증: [확인 방법]
```

## 5. 파일 헤더 주석

새 소스 파일을 만들 때 첫 줄에 파일명과 한 줄 영어 설명을 작성한다.

```python
# train.py: Entry point for training loop with config-based setup.
```

```typescript
// auth-context.tsx: Context provider for managing user authentication state.
```

- `'use client'`, `'use server'`, shebang 등 필수 지시문 바로 아래에 위치한다.
- 설정 파일(`*.config.ts`, `pyproject.toml` 등)은 생략한다.

## 6. 폴더별 파일 배치

| 폴더 | 배치 기준 |
|---|---|
| `src/` | 재사용 가능한 모듈·클래스·함수. 프로토타입 코드는 두지 않는다 |
| `notebooks/` | 탐색·시각화·프로토타입. 정제가 완료되면 `src/` 로 이동한다 |
| `scripts/` | 파이프라인·전처리·배치 실행. CLI 진입점 역할 |
| `experiments/` | 실험 실행 기록 및 메타데이터. 코드가 아니라 기록 중심 |
| `tests/` | `src/` 의 단위·통합 테스트. `notebooks/` 는 테스트 대상이 아니다 |
| `configs/` | YAML·JSON·TOML 설정 파일. 하드코딩 값을 여기로 분리한다 |
| `outputs/` | 모델 파일, 그래프, 예측 결과. 소스 추적 가능해야 한다 |

## 7. 실험 관리

`experiments/` 에 실험을 기록할 때 다음을 따른다.

- 폴더명 형식: `YYYYMMDD_실험명` (예: `20260615_baseline-lr-sweep`)
- 각 실험 폴더에 `README.md` 를 두어 목적, 설정, 결과를 요약한다.
- 실험 설정은 `configs/` 에서 참조하거나 복사한다.
- 결과 파일은 `outputs/` 에 저장하고 실험 `README.md` 에 경로를 기록한다.
- 재현 불가능한 실험은 기록하지 않는다. 재현 방법을 `README.md` 에 명시한다.

## 8. 테스트 실행

코드를 수정했으면 완료 전에 반드시 테스트를 실행한다.

- `pytest`, `npm test` 등 프로젝트 테스트 명령을 실행한다.
- 테스트가 없으면 최소한 빌드·임포트 오류가 없는지 확인한다.
- 통과 여부를 보고하고, 실패하면 수정 후 재실행한다.
- 사용자가 "완료", "끝", "다 됐어" 신호를 보내기 전에 먼저 실행한다.

## 9. 커밋 단위

하나의 논리적 변경이 완료되면 커밋한다.

- "하나의 문장으로 커밋 메시지를 쓸 수 있는가." 가능하면 커밋한다.
- 관련 없는 변경을 하나의 커밋에 묶지 않는다.
- 커밋 메시지는 한국어로 작성한다.

## 10. 오류 읽기

오류가 발생하면 실제 메시지를 읽는다.

- 전체 오류 메시지와 스택 트레이스를 읽는다.
- 가정으로 "흔한 수정"을 먼저 적용하지 않는다.
- 원인이 불명확하면 print·log 를 추가해 상태를 확인한 후 수정한다.
